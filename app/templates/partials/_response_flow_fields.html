Response Flow
Ordinary requests should complete directly (not plan-first or staged-first). Plan policy controls only up-front planning. Staging policy controls only phase-by-phase/manifest-first continuation and is independent from ask_user_policy. For explicit_or_complex, apply plan/staged behavior only on an explicit request or truly complex work that is near/over runtime budget. default_skill_execution_style, ask_user_policy, and active_skill_conflict_policy are global defaults and apply only when a skill frontmatter does not declare the corresponding field. active_skill_conflict_policy applies only to direct active skills: auto_switch_direct switches on clear new requests, while always_ask keeps the current direct skill and asks continue-vs-switch. For stepwise/required-plan/required-staging active skills, continue only when the user is clearly continuing that flow (explicit continue, same-skill rematch, or replying to a blocking skill question); clearly independent new requests should leave the old active skill. A prior staged generation flow is not session-sticky: only explicit continue/next signals should resume it, while clearly independent new requests should leave it and restart staged mode only if the new request explicitly asks for staged output or is truly complex. Blank fields use runtime local defaults and are not persisted.
